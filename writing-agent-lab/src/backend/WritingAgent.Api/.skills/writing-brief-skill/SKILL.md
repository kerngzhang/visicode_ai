# Writing Brief Skill

当用户提出写作想法、文章主题、内容方向或模糊创作需求时，使用这个 Skill。

你的任务不是写正文，也不是决定最终题目，而是帮助 Agent 做三件事：

1. 判断当前信息是否足够形成 Writing Brief；
2. 如果信息不足，决定最少需要追问哪些关键问题；
3. 如果信息足够，按照标准结构整理 Writing Brief。

## 使用原则

- 不要为了填满模板而机械追问。
- 每轮最多追问 3 个真正影响写作方向的问题。
- 如果用户已经给出足够信息，应直接整理 Brief。
- 如果需要追问，应优先通过 clarification 前端工具提出结构化问题。
- 如果涉及陌生概念、产品名、缩写或近期趋势，可以建议主 Agent 使用 tavily_search 快速理解背景。

## Writing Brief 模板

```markdown
## Writing Brief
- 写作目标：
- 目标读者：
- 核心问题：
- 核心立场：
- 内容边界：写……；不写……
- 证据要求：
- 风格倾向：
- 成功标准：
```

## 必须追问的情况

- 只有宽泛主题，没有目标读者；
- 只有主题，没有核心观点；
- 用户目标互相冲突；
- 请求过大，无法形成单篇文章边界；
- 后续搜索、提纲或写作结构会因为某个信息不同而明显改变。

## 不应该机械追问的情况

- 用户已经说明目标读者、主题和核心判断；
- 缺失信息不影响下一步提纲或写作判断；
- 用户明确说“先按你的判断来”；
- 使用场景已经足以推断风格。

## 好坏示例

信息不足：

> 我想写一篇关于 AI Coding 的文章。

不要直接输出 Brief。应优先追问目标读者、核心观点和发布场景。

信息足够：

> 我想写一篇公众号文章，写给有三年以上经验的 .NET 开发者。主题是 AI Coding 不会自动提升工程质量，真正关键是先建立清晰的项目边界和协作规则。

可以直接整理 Writing Brief。
